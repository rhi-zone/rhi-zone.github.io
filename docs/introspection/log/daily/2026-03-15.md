**Normalize view subcommand restructuring — the enum inheritance pattern** — Session d9d58041 (111 msgs, 15:50–19:05 and continuing on Mar 16) tackled the long-deferred view API refactoring. The conversation was combative in the best way: user and agent circling around why `view` and `view list` exist as distinct operations, whether filters should prune or just select, how to handle multiple matches, and what the semantic difference is between `normalize view src/ --kind function` and `normalize view list`. Key insight: the CLI had conflated "filter" (returns a subset) with "query" (returns specific nodes by name/path). The resolution: `view list` returns all matching symbols of a given kind (flat Vec); `view` returns a single selected node's structure tree. On multiple matches, `view` errors with helpful context on why there are duplicates (e.g., TypeScript type vs const with same name). Filters like `--kind` work uniformly across both. The API design debt surfaced again: an enum return `ViewOutput` shouldn't exist — the type system itself should enforce the distinction. By session end, a plan was written to implement the refactored API and dispatch or test locally. Continuation on Mar 16 shifted to command naming (renaming graph subcommands from domain-specific like `refs/deps` to general like `incoming/referenced-by`), session introspection (`sessions messages --role tool`), and a discovered version mismatch in server-less (0.4.6 depends on 0.4.1-alpha.2 → 0.4.2). This is healthy. The user treating agent overconfidence as a signal to revisit CLAUDE.md suggests the documentation is being actively used as a control surface, not just reference material.

Friction often surfaces only when trying to fix related problems; the user is extracting it as data to feed back into core design.

By session end, the main session has already handoff'd and moved on.

The agent is expected to make small decisions (monitoring, handoffs) without user direction.

Large implementation sessions routinely hit the limit, requiring re-summaries. This is not a blocking issue but a cost: each recovery turn burns tokens on re-grounding.
