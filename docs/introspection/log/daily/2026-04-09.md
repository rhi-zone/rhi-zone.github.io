## Design Work

### Platform Vision Refinement

The conversation evolved from "implement lib/reactive signaling" into a fuller platform picture:

- **Not a card-management system.** The portable app platform is a **script sandbox runtime** where each card is a Lua script with a top-level function `(world_state, input) -> (world_state, output)`. The script can access capabilities (LLM, render, file I/O, etc.) and read/write persistent world state (ECS store).

- **Bootstrap tooling (self-referential).** The platform includes a best-in-class **card editor**, **lorebook editor**, and **chat viewer** — and these tools themselves are cards (scripts) running inside the platform. The entire editor stack is bootstrapped from Lua.

- **Flat metadata, not hierarchies.** Bookmark/context organization is not folders but **arbitrary key-value metadata** per card/entry. The primitive is metadata, tags are one use case, but the system is data-driven and composable.

- **CCv2 compat (read/write).** Cards can contain SillyTavern-compatible fields (tEXt chunks with JSON) and arbitrary Crescent extensions (in the same tEXt metadata). On import, a configurable mutate script can transform the card structure. On export, both representations persist — same card, two views.

- **Write-through capability.** A card can ask for a live reference to a section of itself and write back changes; the host patches the PNG metadata on disk. Enables in-place editing of card definitions inside the runtime.

- **Presets as code chunks.** "New character card" doesn't mean "use a form" — it means "copy this Lua snippet into the script, edit, commit." A preset is a composable unit of starter code, not a visual builder.

## Technical Achievements

### Typechecker Work

- **lib/ecs, lib/platform, lib/caps/* fixes**: Resolved recursive type annotations, method annotation conflicts, and narrowing scopes. Core issue: the typechecker doesn't narrow variables to non-nil when passed as function arguments, and `setmetatable` returns an opaque type. Workarounds documented.
- **lib/log discovery**: `any?` as the last parameter in a function annotation corrupts struct field resolution. Switched to `any` without `?`; bug filed in TODO.md.

### Specification and Testing

- **lib/mdast**: From 20 test failures to **652-fixture CommonMark compliance** (100% on 7 categories: ATX headings, code blocks, paragraphs, blank lines, soft breaks, textual content, precedence). Gaps in reference-style links and HTML blocks acknowledged.
- **lib/jsonrpc**: Full JSON-RPC 2.0 spec (batch requests, error codes, both request and notification paths).
- **lib/uuid**: RFC 9562 (v7 with monotonic sequence under same-millisecond bursts; v4 with random generation fallback tiers).
- **lib/tui**: 57 assertions covering flex layout, border styles, alignment, word-wrap, styled composition.

### Regression Fixes

**lib/ljsocket regression**: Commit `66468e9` fixed a memory leak by freeing `addrinfo` early and storing values as Lua instead of FFI pointers. Broke `bind` and `connect` (both indexed `res.addrinfo.ai_addr`). Fixed by re-calling `getaddrinfo` on demand inside those methods.

## Interesting Observations

1. **Self-referential design.** The card editor, lorebook editor, and chat UI are themselves scripts running inside the platform — a bootstrapping pattern that requires careful API design (caps + sandbox, not introspection).

2. **Write-through metadata.** Asking for a live reference to a card section and patching PNG tEXt chunks in-place is a neat efficiency win over parse-rebuild-write cycles.

3. **CCv2 as a transport, not a schema.** Rather than "convert to our format", the platform reads/writes SillyTavern cards as-is, preserving both CCv2 fields and adding extensions alongside. Interop as coexistence, not migration.

4. **Metadata as the missing primitive.** The conversation kept circling back to organization — bookmarks, lorebook entries, card search. The unifying concept isn't "folders" or "tags" but **structured metadata per entity**. This is likely the "gap visible after reboot" the user hinted at.

5. **Library-first, app-later.** Instead of building the full platform, the session focused on the foundational libraries. This approach surfaces assumptions (e.g., ECS use cases, whether "orchestration" is the right abstraction) before they calcify.
