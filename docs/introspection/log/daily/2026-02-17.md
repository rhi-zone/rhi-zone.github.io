## What Actually Happened

### The Big Swing: SugarCube Parser Migration (reincarnate)

Completed a major architectural refactor: **replacing the hand-rolled Pratt parser (~760 lines of custom code) with oxc_parser**. This was the kind of decision that required multiple sessions of debate and clarification (sessions 48b0c4a2 exploring "should we extend the parser?") before committing to the right approach.

Commits show the migration was not just a swap—it required:

- **HTML entity handling** - Replaced hand-rolled decoder with `html-escape` crate (supports full HTML5 spec: 2,231 named entities + numeric forms)
- **Context-aware preprocessing** - Keywords shouldn't trigger inside identifiers (`_is`, `$toString`) or property names (`:` following)
- **SugarCube-specific semantics** - DoL uses `<<case N>>` ranges (not standard JS), trailing commas in case values, template literal preprocessing
- **Statement parsing** - Extended to handle `for`, `throw`, `switch`, `while` (not just expressions)
- **DOM macro classification** - Link, replace, append, etc. take CSS selectors/durations (Raw args), not JS expressions

The migration reduced parse errors from **974 → 0 in TRC test suite** and **290 → 4 in DoL** (remaining 4 are DoL-specific `<<case lte N>>` syntax, not standard SugarCube).

Key lesson added to CLAUDE.md: *Don't hand-roll what a library does. Use crates for standards.*

### Save System Architecture (game persistence)

Session 929f804b spent significant time on save strategy philosophy:

- Twine games currently have **no continuous saving** — progress lost on close without manual save
- Harlowe uses `sessionStorage` (ephemeral), no slots, no FS saves OOTB
- Discussion moved from "how many rolling buffers?" to **why ask those questions at all?** — the answer is the architecture should support pluggable backends (multiSlot + rollingSnapshots for example; both sessionStorage + localStorage + FS when available)
- Consensus: Provide "clean default" impls for deployers, not rigid constraints
