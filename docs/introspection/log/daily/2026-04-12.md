## Key Work: Crescent

### Platform & App Architecture

Early morning recovery from network failure (06:49) — large continuation with ~51k output tokens. Pushed through several design questions around app vendoring, tarball structure, and capability-based security.

**Vendoring strategy settled:**
- First-party apps (card, library, adapters) vendor ALL dependencies (base64, json, reactive, lib.* modules) into their tarballs for portability
- Apps distributed as self-contained units — no external library dependencies required
- Users with "full" Crescent distribution can optionally use system libraries; first-party suite always bundles to prevent silent breakage
- Security implication: shipping source code in PNGs is acceptable because capability grants are explicit (platform validates required caps before running)

**Convention clarified:** Vendored deps live under `_vendored/` inside app tarball root (first-party convention, not mandatory for third-party).

### Design Questions Explored

**Tarball containment & platform boundaries:**
- Apps are self-contained tarballs (not directly executable — require platform loading)
- No Lua symlinks or path manipulation — standard `require()` with tarball-relative root as namespace
- Host provides fallback `lib.*` modules if app doesn't vendor them
- Solved: where to put platform vs app-specific logic (platform infrastructure in host, app logic vendored)

**Card format & type safety:**
- Questioned whether to use full TypeScript-style typedefs vs untyped JSON
- Pragmatic approach: CCv2 format normalized with known structure, adapters provide interface boundaries
- Version detection: v1 (no `spec` field) vs v2 (structured `spec`)

**Lorebook isolation & SQLite constraints:**
- Initial trigger approach failed (qualified names recursion in temp views)
- Redesigned: structured write helpers + authorizer for DDL protection instead of triggers
- App must include `app_id` in INSERT/UPDATE/DELETE — security question: **how to enforce this?** (deferred to runtime validation in caps.db API)

## Observations

1. **Vendor-everything strategy works for closed ecosystems** — Crescent is LuaJIT-based with minimal stdlib, making tarball self-containment tractable. PNG-bundled source code + capability grants creates interesting distribution/security tradeoff.

2. **Subagent delegation effective for module implementations** — Four parallel library builds with high assertion coverage (76+ assertions each), minimal back-and-forth required once spec was clear.

3. **Design-first prevents rework** — Early conversations around platform boundaries, app packaging, security model prevented mid-implementation pivots. The "structured write helpers" discussion (vs triggers) and vendor strategy both involved live rethinking but converged cleanly.

4. **Crescent's scope is maturing** — From "generalized SillyTavern alternative" to concrete three-app suite (card, library, adapters) with clear boundaries. First-party apps are reference implementations; third-party apps follow same pattern but less constrained.

5. **Metadata/search emerging as ecosystem need** — Both io and fuwafuwa showing interest in "tags" / "metadata" across projects. Likely prep for cross-project knowledge infrastructure.
