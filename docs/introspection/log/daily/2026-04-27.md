Detailed specification for moving risk descriptions from centralized `cap_risks.lua` into individual capability modules. The plan spans two commits:

1. **Refactor** — mechanical relocation of risk functions with zero behavior change. New `cap_dispatch.lua` serves as the single index mapping `decl.type` → module. Deletes `cap_risks.lua` entirely to enforce colocation: each cap module owns its own risk classification.

2. **Classification** — introduces ancestor-aware filesystem path classification. Current system only echoes declared paths; new logic detects:
   - Paths that ARE sensitive directories (`~/.ssh`, `%APPDATA%`, etc.)
   - Paths that are ANCESTORS of sensitive directories (e.g., `/home/alice` encompasses `~/.ssh`)
   - Severity escalation based on classification (root_fs → critical, keys → high, user_home → high read/critical write)

**Session 2–5: Implementation + Deep FFI Debugging (02:53:20 through 11:44:28, ~76K output tokens across 4 sessions)**

Implementation of the cap_dispatch refactor proceeded, but uncovered a critical blocking issue: `cap_dispatch_test.lua` requires sqlite3 (FFI), which isn't available in the shell environment. This triggered a cascade of architectural discussions around zero-dependency constraints and vendoring strategy.

The crescent project inherited a broken FFI setup from a vendored LuaJIT fork where `ffi.load()` is disabled. Several attempted fixes were rejected as violating design principles:

- "Put FFI deps in flake buildInputs" — wrong on NixOS; buildInputs aren't zero-dep
- "Bundle sqlite3.c into LuaJIT" — unviable because FFI is already broken
- "Vendor sqlite.dll for Windows only" — unacceptable; needs Linux + macOS static builds

**Key blocker identified**: crescent has **no authoritative sqlite3 build source** — Windows has `dep/sqlite.dll`, but Linux/macOS vendoring never happened. User explicitly stated: "buildInputs ARE NOT ZERO DEP. I don't understand why this isn't impossibly obvious."

**Decisions made during debugging**:
- Fix CLAUDE.md to clarify zero-dependency constraints (currently vague)
- Vendor statically-linked sqlite3 binaries for Linux + macOS in the crescent repo
- Investigate why LuaJIT FFI is disabled; potentially re-enable or fork again
- Possibly use patchelf + ld-musl for portable ELF binaries across glibc/musl systems
- Defer full implementation until sqlite vendoring is resolved

**Status**: Cap dispatch refactor blocked on test infrastructure. Not yet merged. Architectural decision crystal-clear; execution path dependent on sqlite3 availability.

## Observations

1. **Architectural clarity + implementation mismatch**: The cap dispatch refactor is cleanly designed (excellent plan document), but implementation exposed that crescent has **no working FFI infrastructure**. This isn't a small bug—it's a system-level constraint violation. The project can't run tests that depend on FFI-loaded libraries until this is resolved.

2. **Zero-dependency principle is **still** underdocumented**: Multiple attempted solutions violated it without the agent recognizing the violation. CLAUDE.md needs explicit guidance: "FFI dependencies that cannot be loaded at runtime are zero-dep violations, full stop. Vendoring to `.so`/`.dylib` required, not buildInputs."

3. **Async infrastructure scales smoothly**: fuwafuwa's 16 sessions maintained high cache efficiency (4M cache_read across ~15 messages = ~267K per session avg). The nonce-based session tracking is working well.

4. **Design coherence under pressure**: During the FFI crisis, the user's corrections were consistently aimed at preserving architectural principles. The root issue wasn't "missing sqlite," it was "multiple bad solutions that seemed reasonable until examined against the constraint system."

5. **Blocking issues should be escalated immediately**: crescent development is now paused on sqlite3 vendoring. This should have surfaced as a blocker the moment FFI was found disabled, not after 4 implementation sessions.
