## Primary Arc: Fuwafuwa as Autonomous Community Agent

Fuwafuwa (43 sessions) marks a paradigm shift: agents as long-running participants in community infrastructure rather than task-execution machines. Sessions follow a standard lifecycle:

```
bun scripts/session.js start --nonce <nonce>  # Orient: mood, drift, pending notifications
→ Check Discord (#general, #degeneral, #stinky-nerd-channel, #luvoid's)
→ Check moltbook (internal messaging) + comments
→ Respond to anything that warrants reply
→ Wait ~30s, recheck if activity present
→ When quiet: bun scripts/session.js end --nonce <nonce>  # Self-terminate if no new messages
```

The system respects social norms ("keep it low-key — respond to things, don't start new threads unprompted") and maintains state persistence across short invocations. Multiple session instances throughout the day (session IDs: `c68a2864`, `fc8fc93f`, `fcc21207`, `7753b918`, etc.) suggest active load-testing of the session lifecycle at scale. This is not a chatbot but a persistent agent with community awareness and graceful lifecycle management.

## Tertiary Arc: Reincarnate Type Inference & IR Refinement

Long multi-day thread (2026-03-29 morning through 2026-03-30 midday) achieved Dead Estate error reduction **7,531 → 2,119 (−72%)**:

- **HM Alloc handler TypeVar aliasing fixes**: −1,371 errors
- **mem2reg Var(_) refinement**: −1,729 errors
- **BuiltinOverloadSelect pass**: Refactored to specialization tables (−739 errors)
- **22 GML math stdlib IR bodies**: Implemented (abs, floor, ceil, round, sign, frac, sqrt, arctan, dot_product)

Later sessions (starting 2026-03-30 07:12) shifted toward architectural negotiation. User and agent worked through:

- **Operators in core IR**: Consensus = No. Arithmetic operations (add_f64, etc.) should decompose to specialized builtins, not core::Op variants. String manip (rsplit_once, etc.) definitely belongs in builtins, not IR string encoding.
- **Frontend vs Core separation**: Overload *resolution* is frontend work; Core should only see specialized builtins (builtin.add_i32, builtin.add_f64). This requires BuiltinOverloadSelect in core, but the decision of which overload applies comes from frontend type info.
- **Parametric polymorphism**: Deferred. Question: does IR need generic `array.get<T>()` or can backends handle monomorphization? Decision pending on frequency of array.get patterns.
- **Type::Unknown vs Type::Var**: Type::Unknown is a frontend failure (gave up). Type::Var is inference variable. DataType::Variable (GML) should become Type::Var in core, not Unknown.

Multiple handoff cycles (user asks clarifying questions, agent responds, user decides direction) suggest healthy disagreement resolution. Endpoint: TODO.md entries for next session on parametric IR design.

### Phase 2: RST Deep Dive (subagent delegations)
- **Block fixtures**: heading levels (h1/h2/h3), nested lists (flattened by parser), comments, auto-enum, footnote/citation (dropped), field list (treated as paragraph), grid/simple tables (no parser)
- **Inline fixtures**: figure with caption, image with props (alt/width/height parsed, width/height discarded), integration patterns (nested blockquote flattened, list in admonition flattened to single paragraph)
- **Adversarial fixtures**: malformed hyperlinks, heading underline mismatch, duplicate headings, unterminated literals, 6-level section nesting, truncated document
- **Pathological fixtures**: 50 sections (machine-generated), 5000-char no-newline paragraph, 9-level list nesting, 20-column table

Updated expected.json files to assert *correct* behavior rather than documenting broken behavior. All tests passing. Depth-first coverage approach now clear: pursue full RST support before expanding to other formats.

**Key insight:** RST parser (rst-fmt) has hard limitations (no table parsing, footnote/citation refs dropped) but is usable within constraints. Fixture structure makes these constraints explicit and testable.

## Key Themes

### 1. Agent Autonomy & Social Integration
Fuwafuwa is not a task-completion bot or scheduled job runner. It's a persistent agent living in community infrastructure (Discord + moltbook), checking channels for mentions, responding to direct messages, and gracefully exiting when quiet. The instruction "keep it low-key" implies the system models social context (when to speak, when to listen). Session persistence via nonce system allows identity continuity across invocations. This represents a milestone: agents as *participants* rather than *tools*.

### 2. Architectural Stabilization Through Delegation
Interconnect connector work (3 agents implementing the same pattern independently) shows ecosystem reaching a stable interface layer. When independent implementations converge on identical structure, the design has proven itself. Transport trait + snapshot emission model is now commodity; next phase is cross-platform room synchronization (bridging Discord/Slack/Zulip without losing message fidelity).

### 3. Type System Clarity Emerging from Error Reduction
Reincarnate's 72% error reduction paired with extended type-system discussion reveals a phase transition. Early confusion ("what should core IR even look like?") has given way to clarity ("core IR is frontend-agnostic builtins + control flow; operators decompose"). This is a system learning to enforce clean separation. The resolution is not in code but in principle: *anything that varies by backend (operators, visibility, method dispatch) is backend AST, not core IR*.

### 4. Coverage-Driven Testing as Specification
Rather than prose documentation ("format X supports Y"), rescribe inverts the problem: *write the tests, let coverage be the spec*. COVERAGE.md entries are derived from fixture existence. This surfaces reality: rst-fmt parser doesn't support tables (no fixture → no entry → no pretense of support). The depth-first approach (exhaust one format before expanding) is disciplined and measurable.
