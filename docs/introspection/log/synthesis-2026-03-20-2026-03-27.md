## The prose quality reckoning

Mar 22's pteraworld essay rewriting is a parallel to the code quality reckoning documented in prior syntheses, applied to prose instead of code.

The trigger: a reader described the intent collection essays as "schizoposting." The user investigated, found the concern was about prose that sounded too prescriptive and LLM-generated (universal claims stated as neurological fact, observations immediately unpacked into explanations), and directed a mass rewrite. The core insight — **human-sounding prose frames claims as observations, not certainties** — is a design principle, not a copy-editing rule.

The rewriting covered 44+ essays in 5 parallel batches, each applying the same transformations: soften "your nervous system doesn't distinguish..." to "the nervous system doesn't distinguish...", remove authorial self-consciousness ("I keep writing 'the old configuration'" undercuts the sentence), loosen structured headers into flowing prose, and keep genuine uncertainty while removing hedging-as-softener.

The follow-up was structural: a pre-commit hook for detecting bold heading patterns (Mar 22), and a feature to preserve see-also sections during rewrites (Mar 24). The pattern mirrors the code quality reckoning exactly: identify the problem, mass-fix existing instances, then automate prevention. The reader who said "please don't unschizo it" provided external validation that the style was working — the rewrites needed to clean up LLM artifacts without destroying the voice.

This prose convention will likely propagate. Any ecosystem project with significant prose (legacy, motif docs, existence narrative text) faces the same failure mode: LLMs producing text that sounds authoritative about things that are actually open questions.

**Mar 25: the pivot.** A 13-hour session (130 messages) began with a perf baseline handoff plan but pivoted midway to open design exploration. The critical thread: "pillar 6: discoverability" — minimizing tool calls required to answer a query. The user questioned whether SUMMARY.md maintains value for agents vs humans, explored recursive CLAUDE.md rules, glob-based filename matching, and module-level docs parity. The trajectory is clear: normalize is transitioning from structural analysis (view, rank, rules) to contextual query assembly. The "discoverability pillar" reframes the tool's purpose: not "show me the code structure" but "assemble the context I need in the fewest steps." This is the code intelligence convergence identified in the Mar 17–19 synthesis materializing into a concrete architectural direction.

The Mar 25 session also surfaced a technical crisis: heap corruption in ascent-interpreter's dylib serialization, with discussion of whether to drop dylib support entirely in favor of JIT. This is a dependency-layer decision that could simplify normalize's architecture significantly.

## Crescent: the package manager sprint

Crescent consumed more focused design time this week than any other project, with two massive sessions spanning Mar 25–26.

**The package manager design (88+ messages, 6.5+ hours, 166K output tokens)** covered the full design space:
- Multi-version strategy: `lib/<name>/v1/`, `lib/<name>/v2/` with `init.lua` redirects
- Phantom dependency detection and linting
- Lockfile format with hashes for reproducibility
- Vendoring with diff support for tracking user edits
- Zero build steps — "I don't want a build step, ever"
- JSON serialization benchmarking (pure Lua vs FFI vs Node.js vs lunajson)

The JSON benchmarking thread is revealing: Node.js significantly outperformed pure Lua, and LuaJIT tracing didn't explain the gap. The conclusion — the API itself may be the constraint — suggests crescent's package manager will need to carefully choose its serialization layer rather than assuming LuaJIT speed is sufficient.

**The type system correctness push (Mar 26, 78K output)** ran in parallel with PM design. The user pushed back on schema validation complexity ("why not just do it right"), questioned whether zod-like `Schema<T> -> T` inference requires dependent types (it doesn't, with the right approach), and drove module reorganization (renaming sqlitex, symbol keying for type/check modules).

These two threads — package manager and type system — represent crescent's transition from "feature addition" to "engineering discipline." The prior synthesis identified this transition across normalize, crescent, and reincarnate during the quality reckoning. In crescent, it has now fully arrived. The project is no longer asking "what features do we need?" but "how do we build them correctly?"

The package manager is the week's largest design investment and will likely dominate crescent's next 1–2 weeks of implementation. The design space is fully explored — what remains is execution.

## The curilo reset

Mar 23 brought a complete strategic rework of curilo (the parents project). The gamification layer (XP, streaks, attention mechanics) was stripped away. The replacement thesis: **LLMs solve the #1 problem with teaching — giving every student individualized pedagogical feedback at scale.**

The rework generated a new VISION.md with concrete open design questions: profile format (free-form vs queryable), silent amnesia (version tracking), teacher query architecture (LLM tool-use design), and NL vs dashboard interfaces. The kept elements — lessons, classes, mock exams — are the structural core; everything else is being reimagined around pedagogical modeling rather than engagement metrics.

This is a maturity shift. Gamification is a retention mechanism; pedagogical modeling is a learning mechanism. The rework suggests the user recognized that the prior direction solved the wrong problem (keeping students engaged) rather than the right one (helping students learn specific skills).

## Infrastructure and maintenance threads

**Ascent-interpreter JIT bug (Mar 25)** — A precise root cause was isolated: string comparison operators in JIT compare intern IDs (u32 handles) rather than string content, producing incorrect results when intern order differs from lexicographic order. The fix is straightforward (codegen adjustment) but the discovery method is notable: 1.5 hours from reproduction to root cause. This reflects the ecosystem's growing discipline around bug isolation.

**Existence deployment (Mar 25)** — Cloudflare Pages deployment stabilized after encountering a workerd binary size issue (116 MiB vs 25 MiB limit). The fix involved deploying from the worker/ directory explicitly. Session shows rapid iteration (auth, deploy, verify) with background task notifications used effectively.

**IO introspection infrastructure (Mar 25–26)** — Session data backup strategy finalized. The user compared their stack against Anthropic's developer stack document and identified the ecosystem's differentiator: session log analysis enables understanding "when things break down" — a meta-analytical capability that external case studies cannot provide. Decision made to proceed with systematic backup before writing a guide.
